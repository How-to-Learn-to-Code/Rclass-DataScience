How can computational projects get derailed?

It turns out that computational biologists need to be careful with how they manage their code and data. Leaving everything on your personal/lab computer comes with a lot of risk.

You can reduce the risk of a mishap by housing data on UNC‚Äôs cloud computing service, Longleaf, and putting your code on Github. Both of these provide you with backups that can be accessed from anywhere with the internet.

Good computational practices 101

Computational projects ought to be approached with the same expectations of rigor and reproducibility expected of a bench project. This means that the work needs to be well documented, things need to be properly stored, and everything should be organized clearly enough for someone to reproduce it.

Thankfully, we‚Äôre not the first researchers to run into these problems. A whole suite of tools and services exist to manage these issues:

  • Documenting everything

  • Storing data & getting resources

  • Keeping your R project organized

Suffering from manual version control? git can help.

What is version control exactly? At its core, it‚Äôs a way of keeping track of the changes made to files. Before now, you‚Äôve probably used a system like this:

Go on, git!

git is version control software used to record changes to files. GitHub is a repository hosting service that uses git to help users host/review code and manage projects. You can use git locally (on your own computer) without ever using GitHub. One of the many benefits of also using GitHub is that it stores your repositories (folders with code) in the cloud. On GitHub, it is free to have public repositories, but a paid account is required if you want several private repositories. As a student (or with a .edu email address) you can get GitHub Pro for free.

git/GitHub matter because they:

  • Track every version of every script

  • Publicly document your work

  • Allow for new versions of projects to branch

  • Make it easy to collaborate

Git and GitHub are not:

  • These tools are not Google Drive. If you try to add large files, you‚Äôll run into issues.

  • Similar to the first point, these tools are also not a secure place to store data. If you work with sensitive data of any kind, be careful with what is being publicly shared on GitHub. You can always set repositories to private and share with specific collaborators. You should be able to recreate all intermediate data files, plots, results, etc. using your scripts anyways!

Connecting Longleaf and Github

Github and Longleaf each can be daunting to novice programmers, so lets walk through how to set them up together.

The setup is going to amount to three general steps:

  1. Introduce our Github and Longleaf accounts to each other with something called a SSH key
    • SSH keys (secure shell) are encrypted passwords that link two computer systems
  2. Make our first repository (project) on Github
  3. Learn how to get scripts from Github to Longleaf and update changes we made on Longleaf back to Github

Before we can start that, we‚Äôre going to need to know just a tad about terminals and Bash. These topics could be a whole course onto itself, but in a nutshell you can think of them like this:

Terminals, also called command lines, are text-based software for interacting with your computer. RStudio has a built-in terminal, on the tab next to ‚ÄúConsole‚Äù.

Bash is a type of computer language that understands and carries out the instructions you type in the terminal, usually called ‚Äúshell scripting‚Äù. It‚Äôs very common on Linux and Mac computers. Longleaf uses Linux and working on Longleaf means using a bit of Bash.

Tip

This course isn‚Äôt meant to teach you shell scripting, and you aren‚Äôt expected to fully understand some of the Bash command we‚Äôll run. If you‚Äôd like to learn more, please refer to the following cheat sheets on common bash commands and scripting in bash as helpful resources!

Linking via SSH

Let‚Äôs start by getting the terminal open. In the top left, click View -> Move Focus To Terminal. It should have opened in the panel that also contains tabs for ‚ÄúConsole‚Äù and ‚ÄúBackground Jobs‚Äù.

Next, let‚Äôs find the SSH key associated with your Longleaf account. We‚Äôll run the following bash command in the terminal that we just opened (be aware that terminals are notoriously finicky with copy and paste): cat ~/.ssh/id_rsa.pub

Running this command should print your public SSH key to your screen. Copy this to your clipboard, ensuring the entire key is highlighted (it‚Äôs pretty long!).

Caution

Do NOT forget to add the .pub of the above extension. RSA keys come in pairs, a private and a public version. The private key (i.e. the file that does not have the .pub extension) should NEVER be shared. Sharing your private key will allow malicious actors to interact with services that cache your public key as if they were you!

Now, let‚Äôs go over to Github and set up the link.

  1. Go to profile settings on github and select the ‚Äússh key‚Äù section

  2. Add new key

  3. Name the key (should remind you that this is the key for Longleaf)

  4. Paste the copied ssh key from the cat step above

  5. Create!

With that done, we need to log into Github on Longleaf:

  1. Go back to the RStudio terminal

  2. git config --global user.name "<your-github-username>"

    • Keep the quotes around your username, e.g. git config --global user.name "ramses"

  3. git config --global user.email <your-email>

    • No quotes this time, e.g. git config --global user.email ramses@unc.edu

Great, we‚Äôve connected Longleaf and Github (a herculean task for a beginner programmer!). What we‚Äôll want to do next is make a repository (repo), which you could think of as a self-contained project folder. Let‚Äôs go back to Github:

Note

The hidden directory .git/ and file .gitignore will be covered in the next Section 11.

Organization can greatly improve the experience of coding. It is a way for us to show our future selves some kindness as well as anyone who may maintain our work in the future.

Below is an example of how we may setup directories, or folders, for our project:

.
‚îú‚îÄ‚îÄ .git/
‚îú‚îÄ‚îÄ .gitignore
‚îú‚îÄ‚îÄ data/
‚îú‚îÄ‚îÄ outputs/
‚îú‚îÄ‚îÄ R/
‚îú‚îÄ‚îÄ scripts/
‚îî‚îÄ‚îÄ README.md

Here, we have created 4 new directories: data/, outputs/, R/ and scripts/. These directory names imply something to the viewer about their contents and provide quick navigation.

data/ and outputs/ are perhaps the most self explanatory. We will reserve data for files we may read in for our analysis, and outputs as a place for us to store our results.

R/ and scripts/ may be a bit nuanced. Perhaps the project author will keep helpful reusable R code in the R/ directory, while the scripts/ directory may be analysis workflows that are called from the command line.

A project may, or may not, require this level of granularity. You may choose different directory names as well. However we want to maintain some level of interpretation and do not want to contradict expectations with the files within. Critically thinking about your project structure will ultimately save you time when you return to the project.

Here are some general guidelines to follow:

Organization Do‚Äôs

  • Place files in some sort of relative structure
  • Use descriptive file and directory names
  • Format dates with YYYY-MM-DD

Organization Do Not‚Äôs

  • Place all files at the top level of the repository
  • Stratify files between too many directories
  • Use spaces in file names!

README! ‚Ä¶ Please? üò¢

As the name implies, this file is intended to be read by anyone who happens upon your repository.Think of this as extra documentation that you, as the project owner, may use to communicate aspects of the repository. Include helpful information about your project such as:

  • A brief description of the project
  • What is the scope of the project
    • What problem does it solve?
    • Intended use cases!
    • Examples of what it should not be used for!
  • How to get started or use the project
  • How to contribute or report a bug
  • Project Status: active development, stable, or abandoned

If your repository is being viewed on github.com then top level README.md is displayed like the landing page.

Tip

You may also place README.md within sub directories as well, allowing you to communicate more specific information within.

Here are examples of what you may include in a data/README.md file

  • explanation of what data was collected and how it was processed.

  • documentation of the columns of a .csv file. What the acronyms and data labels mean and how they should be interpreted.

Using git

There are three basic commands you need to push changes from your local repository (your computer) to the remote repository (GitHub): git add, git commit, and git push.

  • .git/ is a hidden directory whose main function is to track your git ... commands. It is generally wise to ignore this directory.

  • .gitignore uses rules to exclude files by name pattern or location

    • Don‚Äôt upload data or large files!

  • Changes need to be staged with git add

    • git add . adds all files not excluded by the .gitignore in the directory

    • git add -i opens an interactive adding session

  • Commit staged changes with a note to your future self

    • git commit -m "Hi future me, this is what I changed"

      Note: Using vim

      If you were to write
      git commit
      and omit the -m "your message here" portion, git will force you to write a message by placing you into an interactive prompt. The default text editor is usually vim which may be tricky to navigate.

      To enter edit mode, press i on your keyboard, and then you may begin writing your message. When you have finished, press ESC to exit your current command mode. To exit vim, press :wq which informs vim to write your changes and quit.

  • Commits are pushed to branches

    • For the main branch, use git push origin main

I don‚Äôt git it!

There is almost a 100% chance you get a git error you do not immediately the meaning of or how to solve it. There is also a nearly 100% chance someone has also run into this issue, and has asked about it on the internet. Often, git will provide an idea of how to fix the issue (e.g. fatal: The current branch project-update has no upstream branch. To push the current branch and set the remote as upstream, use git push --set-upstream origin project-update) but if you are still confused, copy and paste the error message into Google!
